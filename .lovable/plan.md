

## Fix: Settlement Engine Falls Back to `parlay_date` When `commence_time` Is Missing

### Problem
The settlement engine requires `commence_time` on each leg to determine the game date for stat lookups. NBA parlay legs generated by the sweet spots scanner don't include `commence_time`, so the engine returns `pending` with reason `no_game_date` — even though the actual stats exist in `nba_player_game_logs`.

This is why all 32 March 10 parlays are stuck as pending despite games being finished and stats being available.

### Fix

**File: `supabase/functions/auto-settle-ai-parlays/index.ts`**

In the `evaluateLeg` function (around line 440-466), when `commence_time` is null, fall back to the parlay's `parlay_date` instead of bailing out:

```text
Current flow:
  commence_time exists? → use it as targetGameDate
  commence_time null?   → return pending (no_game_date)  ← BUG

Fixed flow:
  commence_time exists? → use it as targetGameDate
  commence_time null?   → use parlay_date as targetGameDate (fallback)
```

Specifically:
1. Pass `parlayDate` (from the parent parlay record) into `evaluateLeg`
2. Replace the `no_game_date` early return with `targetGameDate = parlayDate`
3. Log a warning so we can track when the fallback is used

This is a ~10 line change. After deploying, re-running settlement will grade all 32 pending March 10 parlays.
